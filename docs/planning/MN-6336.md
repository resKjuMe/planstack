## Ziel

Ziel des Projekts ist die vollständige Sicherstellung der Transaktionssicherheit von ActivityLog Einträgen, 
damit keine Änderungen an Datensätzen auftreten können, die trotz LogSchema keinen ActivityLog-Eintrag haben,
einen ActivityLog Eintrag haben, obwohl letztlich keine Änderung an Datensätzen durchgeführt wurde.

## Ansatz

Ein SQL Trigger je Tabelle schreibt Änderungen in eine neu zu erstellende Tabelle `auditLogs`. 
Dies soll für alle Änderungen an Datensätzen passieren, deren Tabellen ein LogSchema besitzen.
Auch Bulk-Inserts sollen in der Tabelle korrekt geloggt werden.

Über die SQL Context Info werden companies_id, users_id, support_users_id und request_id (uuid erstellen) gesetzt.
Der SQL Trigger soll diese Daten auslesen und ebenfalls mit in `auditLogs` rows schreiben.

Erweiterung: Wird ActivityLogCollectior::startLog() aufgerufen, wird zusätzlich `group_id` im SQL Context gesetzt.
Wird ActivityLogCollector::endLog() aufgerufen, dann wird siese `group_id` auf den vorherigen Parent oder NULL wenn nicht vorhanden gesetzt. 
Diese Info soll ebenfalls in `auditLogs` mit aufgenommen werden.

Bulk-Operations müssen so umgeformt werden, dass je Datensatz ein Eintrag in `activityLogs` aufgenommen wird.
Änderungen die innerhalb eines Requests direkt nacheinander erfolgten und die gleiche Tabelle betreffen, werden dabei zusammengefasst.
Eintragsänderungen die laut LogSchema eine PseudoProperty definiert haben, werden für das ActivityLog entsprechend umgeformt.

## Trigger-Erstellung

Die Trigger sollen für alle LogSchema Tabellen automatisch erstellt werden, 
sofern nicht vorhanden, sobald die Migrations ausgeführt werden.
